设计模式

面向对象设计原则

单一职责原则

–Single Responsibility Principle —-类的职责要单一，不能将太多的职责放在一个类中 ——一个对象应该只包含单一的职责，而且该职责被完整的封装在一个类中。(就一个类而言，应该仅有一个引起它变化的原因)

开闭原则

–Open-Closed Principle —-软件实体对拓展是开放的，但对修改是关闭的，即在不修改一个软件实体的基础上去拓展其功能 ——

里氏代换原则

–Liskov Substitution Principle —-在软件系统中，一个可以接受基类对象的地方必然可以接受一个子类对象 ——

依赖倒转原则

–Dependency Inversion Principle —-要针对抽象层编程，而不要针对具体类编程 ——

接口隔离原则

–Interface Segregation Principle —-使用专门的接口来取代一个统一的接口 ——

合成复用原则

–Composite Reuse Principle —-在系统中应该尽量多使用聚合和组合关系，尽量少使用甚至不使用继承关系 ——

迪米特法则

–Law of Demeter —-又称为最少知识原则，一个软件实体与其他实体的引用越少越好，或者说如果两个类不必彼此直接通信，那么这两个类就不应当发生直接的相互作用，而是通过引入一个第三者发生间接交互 ——

jsp

请求重定向和请求转发的区别

请求重定向： response.sendRedirect() 客户端行为

两次请求，所以第一次请求提交的对象不会被保存。地址栏的URL不会改变

请求转发： request.getRequestDispatcher().forward() 服务器行为

一次请求，转发后请求对象会保存，地址栏的URL不会改变

HbaseError:ConnectionLoss for /hbase/hbaseid

Hbase启动成功后，运行hbase shell执行命令的时候抛出了异常如下:

ERROR [main] zookeeper.ZooKeeperWatcher: hconnection-0x3078cac0x0, quorum=slave1:2181,slave2:2181,slave3:2181, baseZNode=/hbase Received unexpected KeeperException, re-throwing exception
org.apache.zookeeper.KeeperException$ConnectionLossException: KeeperErrorCode = ConnectionLoss for /hbase/hbaseid

搜索一番得知：

zookeeper并没有创建/znode 导致hbase无法定位到默认的znode— /hbase 解决方法： 运行zookeeper：./zkCli.sh 创建znode: create /hbase myhbase 现在重启hbase ，hmaster和regionserver正常启动 以上内容来自hbase无法启动，The node /hbase…

创建znode之后，正常启动hbase shell ，无报错。
